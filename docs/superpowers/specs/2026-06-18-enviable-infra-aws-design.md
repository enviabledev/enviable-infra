# enviable-infra — AWS Deployment (Terraform) Design

**Date:** 2026-06-18
**Status:** Approved (design) — pending implementation plan
**Region:** AWS `eu-north-1` (Stockholm)
**Repo:** `enviable-infra` (this repo; manages AWS infra via Terraform)

## 1. Purpose

Provision and manage, entirely with Terraform, the AWS infrastructure that hosts the
**enviable-system** NestJS backend, plus a CI/CD pipeline to deploy it. The
**enviable-web** Next.js frontend is hosted on **Vercel** (not AWS) and is out of
scope for AWS compute — only its one backend-URL env var is documented here.

## 3. Decisions (locked)

| Topic | Decision |
|---|---|
| Frontend subdomain | `portal.enviabletricycle.com` → Vercel |
| Backend hostname | `api.enviabletricycle.com` → AWS EC2 |
| Compute | 1× EC2 **t3.micro**, Amazon Linux 2023, gp3 root, **2 GB swapfile** |
| Backend host model | One subdomain; frontend proxies `/api/*` to backend server-side |
| Database | **RDS PostgreSQL 16, db.t3.micro**, 20 GB gp3, **single-AZ**, private, 7-day backups |
| Redis (sessions) | Run **on the EC2 box** as a Docker container |
| PDF rendering | Chromium baked into the backend image (puppeteer-core) |
| Object storage | **S3 bucket** for generated PDFs + general uploads (versioned, private, encrypted) |
| TLS | **Caddy on the box**, automatic Let's Encrypt, HTTP→HTTPS redirect (no ALB) |
| DNS | **Stays at current provider** — add two records only; zone NOT migrated to Route 53 |
| CI/CD | **GitHub Actions** + GitHub **OIDC** (no long-lived AWS keys) |
| Migrations | `prisma migrate deploy` runs automatically on backend container start |
| Secrets | SSM Parameter Store (SecureString); DB password + SESSION_SECRET Terraform-generated |
| Shell access | **SSM Session Manager** (no inbound SSH) |
| Est. cost | ~$28–32/month |

## 4. AWS Resources (Terraform)

### Network (module `network`)
- Dedicated VPC.
- **1 public subnet**: EC2 + Internet Gateway, full egress (pulls from ECR/SSM). **No NAT gateway** (saves ~$32/mo).
- **2 private subnets** across two AZs: required for the RDS DB subnet group; RDS not publicly accessible.
- Security groups:
  - `ec2-sg`: inbound 80 + 443 from `0.0.0.0/0` (80 needed for ACME HTTP-01 challenge). No SSH.
  - `rds-sg`: inbound 5432 **only** from `ec2-sg`.

### Compute (module `compute`)
- EC2 t3.micro, AL2023, gp3 root volume.
- **Elastic IP** attached → stable address for the `api` DNS record.
- IAM instance role: ECR pull, SSM (Session Manager + Run Command + Parameter Store read), and scoped S3 read/write to the app bucket.
- `user-data`: create 2 GB swapfile, install Docker + Compose, pull secrets, `docker compose up -d`.
- Containers via `docker compose`: **Caddy** (reverse proxy + auto-TLS) → **NestJS backend** (incl. Chromium) + **Redis**.

### Database (module `database`)
- RDS PostgreSQL 16, db.t3.micro, 20 GB gp3, single-AZ, not publicly accessible.
- Automated backups, 7-day retention. DB subnet group across the two private subnets.
- Password generated by Terraform (`random_password`), stored in SSM SecureString.

### Storage (module `storage`)
- S3 bucket: versioning on, public access blocked, SSE encryption.
- Access granted to the backend via the EC2 instance role (no static keys).

### Registry (module `ecr`)
- One ECR repository for the backend image. Lifecycle policy to expire untagged images.

### Secrets (module `ssm`)
- SSM Parameter Store SecureStrings: `SESSION_SECRET` (generated), RDS password (generated),
  `DEFAULT_INITIAL_PASSWORD`, `DATABASE_URL`/`REDIS_URL` (assembled), invoice config vars.
- Deploy step injects them into the container environment at start.

### CI/CD identity (module `cicd`)
- GitHub OIDC provider + IAM role assumable by the GitHub Actions workflows
  (scoped to ECR push, SSM send-command, and Terraform plan/apply).

### Remote state
- Terraform state in an **S3 bucket** with a **DynamoDB lock table**.
- A one-time **bootstrap** (local state) creates the state bucket + lock table before the main stack uses them.

## 5. CI/CD Pipelines (GitHub Actions, OIDC)

- **enviable-system (backend):** push to `main` → build Docker image → push to ECR →
  `aws ssm send-command` runs `docker compose pull && docker compose up -d` on the instance.
  `prisma migrate deploy` runs on container start.
- **enviable-infra (this repo):** PR → `terraform plan`; merge to `main` → `terraform apply`
  (gated by a GitHub environment approval).
- **enviable-web (frontend):** Vercel native Git integration — no AWS pipeline. Only config is the
  `BACKEND_API_URL` env var in the Vercel project.

## 6. DNS records to add (at current provider — apex/MX/DKIM untouched)

| Name | Type | Value |
|---|---|---|
| `api` | A | `<Elastic IP from Terraform output>` |
| `portal` | CNAME | `<target Vercel provides for the project>` |

Adding subdomain records does not affect the apex website, `MX`, or `google._domainkey` DKIM
records. The zone is **not** migrated to Route 53.

## 7. Vercel environment variables (frontend)

| Var | Value |
|---|---|
| `BACKEND_API_URL` | `https://api.enviabletricycle.com` |

The frontend holds no secrets. Session cookies remain same-origin because Next proxies
`/api/*` server-side (browser only ever talks to `portal.…`).

## 8. Cross-repo guard constraints

A session repo guard means each repo is managed autonomously. For **enviable-system** and
**enviable-web**, this work will **only append new files** and will **not edit or move existing
files** without explicit permission.

- **New files to add in `enviable-system`:** `Dockerfile`, `docker-compose.yml`, `Caddyfile`,
  `.github/workflows/deploy.yml` (and any deploy helper scripts).
- **New files in `enviable-web`:** documentation of the Vercel env var only.

### Known checkpoint requiring permission (not a silent assumption)
Running NestJS behind Caddy (TLS-terminating reverse proxy) and serving the Vercel-proxied flow
typically needs Express `trust proxy` enabled and the session cookie's `secure`/`sameSite`
configured. **If these are not already environment-driven in existing backend source**, enabling
them will require **editing an existing file in `enviable-system`** — at which point we stop and
request permission before proceeding. The Docker/Caddy/CI files themselves require no existing-file
edits.

## 9. Cost estimate (eu-north-1, monthly, approx.)

| Item | Cost |
|---|---|
| EC2 t3.micro | ~$7.5 |
| EBS gp3 (root) | ~$2 |
| Elastic IP (public IPv4) | ~$3.6 |
| RDS db.t3.micro | ~$13 |
| RDS storage (20 GB gp3) | ~$3 |
| S3 + ECR | cents |
| **Total** | **~$28–32** |

No NAT gateway, no ALB, no Route 53 zone.

## 10. Out of scope (now)

- Multi-AZ RDS / autoscaling / ALB (revisit on growth).
- Backend S3 integration code (bucket + IAM provisioned now; app wiring is later backend work).
- Moving frontend off Vercel.
- Email (SendGrid) and other backend-feature work.
